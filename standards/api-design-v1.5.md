# API 设计规范 v1.5

## 1. URL 设计

### 1.1 基本规则
- 使用小写字母
- 使用连字符分隔单词：`/api/v1/user-profiles`
- 使用复数名词表示集合：`/api/v1/users`
- 避免过深的嵌套（不超过 3 层）

### 1.2 版本控制
- URL 中包含版本号：`/api/v1/`
- 版本升级时保持向后兼容

## 2. HTTP 方法

| 方法   | 用途     | 幂等性 |
|--------|----------|--------|
| GET    | 查询资源 | 是     |
| POST   | 创建资源 | 否     |
| PUT    | 全量更新 | 是     |
| PATCH  | 部分更新 | 否     |
| DELETE | 删除资源 | 是     |

## 3. 请求与响应

### 3.1 请求格式
- Content-Type: application/json
- 分页参数：page, page_size
- 过滤参数：status, category

### 3.2 响应格式
```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "meta": {
    "page": 1,
    "page_size": 20,
    "total": 100
  }
}
```

### 3.3 错误响应
```json
{
  "code": 40001,
  "message": "参数校验失败",
  "details": [
    {
      "field": "email",
      "message": "邮箱格式不正确"
    }
  ]
}
```

## 4. 状态码使用

- 200：成功
- 201：创建成功
- 204：删除成功（无返回体）
- 400：请求参数错误
- 401：未认证
- 403：无权限
- 404：资源不存在
- 409：资源冲突
- 422：参数校验失败
- 500：服务器内部错误
